# 小程序首页类型数据可配置需求设计文档

## 1. 需求背景

当前小程序首页卖家模式的卡券类型数据是硬编码在 `index.js` 的 `CARD_TYPES_DATA` 常量中的，无法动态配置。需要改为从数据库获取，实现：
- 数据库中有类型就展示，没有就不展示
- 管理后台新增/删除类型后，前端自动同步展示
- 首页仅展示类型名称（不展示折扣）
- 点击类型 → 进入该类型下的所有卡券列表（不展示折扣）
- 点击卡券 → 进入订单提交页面（卡号卡密输入 + 面值选择 + 卖出折扣展示）

## 2. 现有架构分析

### 2.1 数据库三层结构
```
CardType（卡券类型） → CardProduct（卡产品） → CardProductFaceValue（面值明细）
```

- **CardType**：如"电话卡"、"加油卡"、"游戏卡"等大类
- **CardProduct**：如"移动充值卡"、"中石化加油卡"等具体产品
- **CardProductFaceValue**：如100元面值、200元面值，每个面值有独立的收卡折扣(discountRate)和卖卡折扣(buyDiscountRate)

### 2.2 现有API
| 接口 | 说明 |
|------|------|
| `GET /api/card/typeList` | 获取所有启用的卡券类型列表 |
| `GET /api/card/card-product/:id/face-values` | 获取卡产品的回收面值列表 |
| `POST /api/card/calculate` | 计算回收金额 |
| `POST /api/order/create` | 创建订单 |

### 2.3 缺少的API
| 需要的接口 | 说明 |
|------------|------|
| `GET /api/card/type/:typeId/products` | 获取某类型下所有启用的卡产品列表 |

## 3. 页面流程设计

```
首页（类型列表） → 卡券列表页（某类型下的卡产品） → 订单提交页（面值选择+卡号卡密）
```

### 3.1 首页（类型列表）
- 数据来源：`GET /api/card/typeList`
- 展示内容：类型名称、图标
- 不展示折扣信息
- 点击某个类型 → 跳转到卡券列表页

### 3.2 卡券列表页（新增页面）
- 路径：`pages/card/list/list`
- 数据来源：`GET /api/card/type/:typeId/products`
- 展示内容：卡产品名称、图标
- 不展示折扣信息
- 点击某个卡产品 → 跳转到订单提交页

### 3.3 订单提交页（复用现有 detail 页面改造）
- 路径：`pages/detail/detail`
- 数据来源：`GET /api/card/card-product/:id/face-values`
- 展示内容：
  - 卡号输入框
  - 卡密输入框
  - 面值选项列表（每个面值展示卖出折扣 discountRate）
  - 选中面值后展示回收金额
- 提交：`POST /api/order/create`

## 4. 后端API设计

### 4.1 新增接口：获取某类型下的卡产品列表

**URL**: `GET /api/card/type/:typeId/products`

**请求参数**:
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| typeId | Integer | 是 | 卡券类型ID（URL路径参数） |

**响应示例**:
```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "name": "移动充值卡",
      "icon": "📱",
      "isHot": 1,
      "sort": 0
    },
    {
      "id": 2,
      "name": "联通充值卡",
      "icon": "📱",
      "isHot": 0,
      "sort": 1
    }
  ]
}
```

**逻辑**:
1. 查询 `card_product` 表中 `card_type_id = typeId` 且 `status = 'ACTIVE'` 的记录
2. 按 `sort ASC` 排序
3. 只返回 id、name、icon、isHot、sort 字段

### 4.2 现有接口改造：订单创建支持 cardProductId

现有 `POST /api/order/create` 已支持 `cardProductId` 参数，无需修改。

## 5. 前端改造设计

### 5.1 首页改造（pages/index/index）

**改动点**:
1. 删除 `CARD_TYPES_DATA` 硬编码常量
2. `loadSellerCardTypes()` 改为调用 `GET /api/card/typeList` 获取数据
3. 类型列表只展示名称和图标，不展示折扣
4. 点击类型跳转到 `pages/card/list/list?typeId=xxx&typeName=xxx`

### 5.2 新增卡券列表页（pages/card/list/list）

**页面结构**:
- 顶部导航栏：显示类型名称
- 卡产品网格列表：展示卡产品名称和图标
- 点击卡产品跳转到 `pages/detail/detail?cardProductId=xxx&cardTypeId=xxx`

**文件**:
- `pages/card/list/list.js`
- `pages/card/list/list.wxml`
- `pages/card/list/list.wxss`
- `pages/card/list/list.json`

### 5.3 订单提交页改造（pages/detail/detail）

**改动点**:
1. 接收参数从 `id`（cardTypeId）改为 `cardProductId` + `cardTypeId`
2. 调用 `GET /api/card/card-product/:id/face-values` 获取面值列表
3. 面值选项展示：每个面值显示面值金额 + 卖出折扣（如"100元 · 95折"）
4. 选中面值后展示回收金额
5. 提交订单时传入 `cardProductId`

### 5.4 API配置新增（config/api.js）

```javascript
card: {
  // ...existing
  typeProducts: `${API_BASE}/card/type`,  // 某类型下的卡产品列表（需拼接 /:typeId/products）
}
```

### 5.5 app.json 新增页面路由

```json
"pages/card/list/list"
```

## 6. 数据库变更

无需数据库变更，现有表结构已满足需求。
