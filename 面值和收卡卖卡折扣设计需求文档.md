# 面值和收卡卖卡折扣设计需求文档

## 一、现状与问题

### 1.1 当前数据结构

```
CardType (卡类型，如"话费充值卡")
  └── CardProduct (卡产品，如"中国移动充值卡")
        ├── faceValues: [20, 30, 50, 100]   -- 所有面值共用
        ├── discountRate: 0.93              -- 所有面值共用收卡折扣
        ├── buyDiscountRate: 0.95           -- 所有面值共用卖卡折扣
        ├── isHot: 0/1                      -- 卡产品级别
        └── isSaleable: 0/1                 -- 卡产品级别
```

### 1.2 存在的问题

| 问题 | 说明 |
|------|------|
| **面值折扣不合理** | 同一卡产品下，20元面值和1000元面值使用相同的收卡/卖卡折扣率，不符合实际业务——大面值通常折扣更低（回收风险更高），小面值折扣更高（流通性好） |
| **isHot/isSaleable 粒度过粗** | "是否热门"和"是否销售"标记在卡产品级别，但实际业务中可能100元面值热门而500元面值不热门，100元面值可售而5000元面值暂停销售 |
| **数据冗余顾虑** | 若为每个面值建一行记录，会导致数据量暴增（如京东E卡有21种面值，8种卡产品 × 21 = 168行），管理复杂 |

---

## 二、设计方案

### 2.1 核心思路：面值明细子表

新增 `card_product_face_value`（卡产品面值明细表），将面值从 `card_product` 的 JSON 字段拆分为独立子表行，每行代表一个面值及其专属的折扣率和状态。

**优势：**
- 每个面值可独立设置收卡折扣、卖卡折扣、是否热门、是否销售
- 数据量可控：当前全量卡产品约30种，面值总计约200条，完全可接受
- 查询灵活：可按面值维度筛选、统计、排序
- 前后端一致：面值数据统一由后端管理，小程序从 API 动态获取

### 2.2 数据库设计

#### 新增表：`card_product_face_value`

```sql
CREATE TABLE `card_product_face_value` (
  `id` INT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `card_product_id` INT NOT NULL COMMENT '所属卡产品ID',
  `face_value` DECIMAL(10,2) NOT NULL COMMENT '面值金额',
  `discount_rate` DECIMAL(3,2) NOT NULL DEFAULT 0.95 COMMENT '收卡折扣率',
  `buy_discount_rate` DECIMAL(3,2) NOT NULL DEFAULT 0.95 COMMENT '卖卡折扣率',
  `is_hot` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否热门: 1-是, 0-否',
  `is_saleable` TINYINT(1) NOT NULL DEFAULT 1 COMMENT '是否销售: 1-是, 0-否',
  `sort` INT NOT NULL DEFAULT 0 COMMENT '排序（升序）',
  `status` VARCHAR(10) NOT NULL DEFAULT 'ACTIVE' COMMENT '状态: ACTIVE-启用, DISABLED-禁用',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_card_product_id` (`card_product_id`),
  UNIQUE KEY `uk_product_face` (`card_product_id`, `face_value`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='卡产品面值明细表';
```

#### `card_product` 表变更

```sql
-- 移除已拆分到子表的字段
ALTER TABLE `card_product` DROP COLUMN `face_values`;
ALTER TABLE `card_product` DROP COLUMN `discount_rate`;
ALTER TABLE `card_product` DROP COLUMN `buy_discount_rate`;
ALTER TABLE `card_product` DROP COLUMN `is_hot`;
ALTER TABLE `card_product` DROP COLUMN `is_saleable`;
```

> **注意：** `card_product` 表保留 `name`、`cardTypeId`、`sort`、`status` 等卡产品级别的公共属性。折扣率、是否热门、是否销售下沉到面值明细级别。

### 2.3 数据迁移

从现有 `card_product.face_values` 和 `discount_rate`/`buy_discount_rate`/`is_hot`/`is_saleable` 迁移数据到新表：

```sql
-- 数据迁移：将每个卡产品的面值展开为子表行
-- 需要通过脚本执行（JSON数组展开SQL较复杂，建议用Node.js脚本迁移）

-- 迁移逻辑伪代码：
-- FOR EACH card_product:
--   FOR EACH faceValue IN card_product.faceValues:
--     INSERT INTO card_product_face_value
--       (card_product_id, face_value, discount_rate, buy_discount_rate, is_hot, is_saleable, sort)
--     VALUES
--       (card_product.id, faceValue, card_product.discount_rate, card_product.buy_discount_rate, card_product.is_hot, card_product.is_saleable, index)
```

**迁移脚本要点：**
1. 读取所有 `card_product` 记录
2. 解析每条记录的 `face_values` JSON 数组
3. 为每个面值生成一条 `card_product_face_value` 记录，继承卡产品级别的折扣率和状态
4. 迁移完成后执行 `ALTER TABLE` 删除旧字段

### 2.4 数据层级关系（变更后）

```
CardType (卡类型，如"话费充值卡")
  └── CardProduct (卡产品，如"中国移动充值卡")
        ├── name, sort, status          -- 卡产品级公共属性
        └── CardProductFaceValue[] (面值明细)
              ├── faceValue: 20         -- 面值金额
              ├── discountRate: 0.95    -- 20元面值专属收卡折扣
              ├── buyDiscountRate: 0.97 -- 20元面值专属卖卡折扣
              ├── isHot: 1              -- 20元面值是否热门
              ├── isSaleable: 1         -- 20元面值是否可售
              └── sort: 0
              ├── faceValue: 100
              ├── discountRate: 0.93    -- 100元面值收卡折扣更低
              ├── buyDiscountRate: 0.95
              ├── isHot: 0
              ├── isSaleable: 1
              └── sort: 1
              ├── faceValue: 500
              ├── discountRate: 0.90    -- 500元面值折扣更低
              ├── buyDiscountRate: 0.93
              ├── isHot: 0
              ├── isSaleable: 0         -- 500元面值暂停销售
              └── sort: 2
```

### 3.2 变更接口

#### 3.2.1 新增卡产品

```
POST /api/admin/card-product/create
```

**变更内容：** 请求体移除 `faceValues`、`discountRate`、`buyDiscountRate`、`isHot`、`isSaleable` 字段，新增卡产品时仅创建卡产品基本信息（名称、卡类型、排序、状态），面值明细通过 `batch-save` 接口单独管理。

**变更后请求参数：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| cardTypeId | Integer | 是 | 所属卡类型ID |
| name | String | 是 | 卡产品名称 |
| sort | Integer | 否 | 排序，默认0 |
| status | String | 否 | 状态，默认ACTIVE |

#### 3.2.2 卡产品列表

```
GET /api/admin/card-products/all
```

**变更内容：** 列表不再展示面值、折扣率、是否热门、是否销售列（这些信息下沉到面值明细），改为展示"面值数量"列，点击可展开查看面值明细。

**新增查询参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| isHot | Integer | 否 | 筛选：是否有热门面值（1=是/0=否） |
| isSaleable | Integer | 否 | 筛选：是否有可售面值（1=是/0=否） |

**响应变更：** 每条卡产品记录新增 `faceValueCount`（面值数量）字段，移除 `faceValues`、`discountRate`、`buyDiscountRate`、`isHot`、`isSaleable` 字段。

#### 3.2.3 更新卡产品折扣（废弃）

```
POST /api/admin/card-product/update-discount  ← 废弃
```

> 折扣率已下沉到面值明细级别，该接口不再使用。面值折扣的修改通过 `batch-save` 接口实现。

### 3.3 交易流程接口变更

#### 3.3.1 回收下单

当前回收金额计算使用 `CardType.discountRate`，变更为使用 `CardProductFaceValue.discountRate`：

```
回收金额 = 面值 × CardProductFaceValue.discountRate
```

**校验逻辑：**
1. 根据卡产品ID + 面值查询 `card_product_face_value` 记录
2. 校验该面值 `status=ACTIVE` 且所属卡产品 `status=ACTIVE`
3. 使用该面值的 `discountRate` 计算回收金额

#### 3.3.2 购买下单

当前购买价格计算使用 `CardType.buyDiscountRate`，变更为使用 `CardProductFaceValue.buyDiscountRate`：

```
购买价格 = 面值 × CardProductFaceValue.buyDiscountRate
```

**校验逻辑：**
1. 根据卡产品ID + 面值查询 `card_product_face_value` 记录
2. 校验该面值 `isSaleable=1` 且 `status=ACTIVE` 且所属卡产品 `status=ACTIVE`
3. 使用该面值的 `buyDiscountRate` 计算购买价格

```

└──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

**字段校验：**

| 字段 | 校验规则 |
|------|---------|
| 面值 | 必填，正数，同一卡产品下不可重复 |
| 收卡折扣 | 必填，0~1之间，保留两位小数 |
| 卖卡折扣 | 必填，0~1之间，保留两位小数 |
| 是否热门 | 下拉选择：是/否 |
| 是否销售 | 下拉选择：是/否 |
| 状态 | 下拉选择：启用/禁用 |

### 4.4 批量保存逻辑

1. 用户在弹窗中增删改面值行
2. 点击"保存"时，前端收集所有面值数据，调用 `batch-save` 接口
3. 已有记录传 `id`（更新），新增记录不传 `id`，删除的记录传 `deleted: true`
4. 后端在事务中执行：新增 → 更新 → 删除，保证数据一致性

## 六、Sequelize 模型设计

### 6.1 新增模型：CardProductFaceValue

```javascript
// src/models/CardProductFaceValue.js
const { DataTypes } = require('sequelize');
const sequelize = require('../config/database');

const CardProductFaceValue = sequelize.define('CardProductFaceValue', {
  id: {
    type: DataTypes.INTEGER,
    primaryKey: true,
    autoIncrement: true
  },
  cardProductId: {
    type: DataTypes.INTEGER,
    allowNull: false,
    field: 'card_product_id',
    comment: '所属卡产品ID'
  },
  faceValue: {
    type: DataTypes.DECIMAL(10, 2),
    allowNull: false,
    field: 'face_value',
    comment: '面值金额'
  },
  discountRate: {
    type: DataTypes.DECIMAL(3, 2),
    allowNull: false,
    defaultValue: 0.95,
    field: 'discount_rate',
    comment: '收卡折扣率'
  },
  buyDiscountRate: {
    type: DataTypes.DECIMAL(3, 2),
    allowNull: false,
```

## 七、数据迁移方案

### 7.1 迁移步骤

1. **创建新表** `card_product_face_value`
2. **执行数据迁移脚本**（Node.js），将现有 `card_product.face_values` 展开为子表行
3. **验证迁移数据**，确认面值数量和金额一致
4. **部署新代码**（模型、服务、路由、控制器、前端）
5. **删除旧字段** `ALTER TABLE card_product DROP COLUMN ...`
6. **更新小程序端**，从 API 动态获取面值数据

### 7.2 迁移脚本核心逻辑

```javascript
const CardProduct = require('./src/models/CardProduct');
const CardProductFaceValue = require('./src/models/CardProductFaceValue');

async function migrate() {
  const products = await CardProduct.findAll();
  for (const product of products) {
    const faceValues = product.faceValues || [];
    for (let i = 0; i < faceValues.length; i++) {
      await CardProductFaceValue.create({
        cardProductId: product.id,
        faceValue: faceValues[i],
        discountRate: product.discountRate,
        buyDiscountRate: product.buyDiscountRate,
        isHot: product.isHot,
        isSaleable: product.isSaleable,
        sort: i,
        status: product.status
      });
    }
  }
  console.log('迁移完成');
}
```

## 八、前后端数据一致性保障

### 8.1 核心原则

| 原则 | 措施 |
|------|------|
| **单一数据源** | 面值数据统一存储在 `card_product_face_value` 表，前端（管理后台+小程序）均从 API 获取，禁止硬编码 |
| **事务保证** | 批量保存面值明细使用数据库事务，保证新增/更新/删除的原子性 |
| **接口校验** | 交易下单时，后端根据卡产品ID+面值查询子表获取折扣率，不信任前端传入的折扣率 |
| **缓存清理** | 面值数据变更后，清除相关 Redis 缓存（如有） |

### 8.2 交易防篡改

回收/购买下单时，后端必须：

1. 根据请求中的 `cardProductId` + `faceValue` 查询 `card_product_face_value` 表
2. 校验该面值记录存在且 `status=ACTIVE`
3. 购买场景额外校验 `isSaleable=1`
4. 使用数据库中的 `discountRate`/`buyDiscountRate` 计算金额，**不使用前端传入的折扣率**
5. 校验前端传入的金额与后端计算金额一致
