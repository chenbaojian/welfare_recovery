# 福利回收微信小程序 - 我是买家 需求文档

## 一、项目概述

### 1.1 功能背景

当前福利回收小程序仅支持"我是卖家"（卖家）模式，用户可以将闲置卡券提交回收变现。为拓展业务场景，现新增"我是买家"模式，让用户可以购买平台已回收且可售的卡券，实现卡券的二次流通，形成"回收→上架→购买"的完整闭环。

### 1.2 功能目标

- 支持两种用户角色模式切换：**我是买家**（当前模式）与 **我是买家**（新增模式）
- 买家模式下，首页展示可销售的卡券种类，用户可浏览、查看详情并下单购买
- 卡券数据来源于回收订单中状态为 `SUCCESS`（已完成）且未被售出的卡券
- 保证前后端数据一致性，避免超卖、重复购买等问题

### 1.3 业务闭环

```
卖家提交卡券 → 管理员审核完结(SUCCESS) → 卡券上架可售 → 买家浏览下单购买 → 卡券售出标记
```

## 二、用户角色与模式切换

### 2.1 双角色模式定义

| 模式 | 角色名称 | 说明 | 核心操作 |
|------|----------|------|----------|
| 我是卖家 | 卖家 | 当前已有模式，用户将闲置卡券提交回收变现 | 选择卡券类型→输入卡号卡密→提交回收订单 |
| 我是买家 | 买家 | 新增模式，用户购买平台已回收的可售卡券 | 浏览可售卡券→查看卡券详情→下单购买 |

### 2.2 模式切换设计

#### 2.2.1 切换入口

在首页顶部区域设置模式切换控件，采用 **Tab切换** 形式：

```
┌─────────────────────────────────────┐
│   [我是卖家]    [我是买家]           │  ← 顶部Tab切换栏
├─────────────────────────────────────┤
│                                     │
│   （当前模式对应的首页内容）          │
│                                     │
└─────────────────────────────────────┘
```

#### 2.2.2 切换规则

| 规则 | 说明 |
|------|------|
| 默认模式 | 用户首次进入默认为"我是卖家"模式 |
| 模式持久化 | 用户选择的模式通过本地缓存（`wx.setStorageSync`）持久化，下次打开小程序自动恢复上次选择 |
| 切换即时生效 | 切换模式后首页内容立即刷新，无需重新加载页面 |
| 登录状态共享 | 两种模式共享同一登录状态和实名认证状态，切换模式不需要重新登录 |
| 订单隔离 | 卖家订单（回收订单）和买家订单（购买订单）分别展示，互不干扰 |

#### 2.2.3 切换交互

| 交互动作 | 系统响应 |
|----------|----------|
| 点击"我是买家"Tab | 首页内容切换为可售卡券种类展示，Tab高亮切换 |
| 点击"我是卖家"Tab | 首页内容恢复为原有卡券回收分类展示，Tab高亮切换 |
| 模式切换时未登录 | 弹出登录引导弹窗，登录后继续当前模式 |
| 模式切换时未实名 | 弹出实名认证引导弹窗，认证后继续当前模式 |

### 3.2 首页数据展示规则

#### 3.2.1 卡券种类展示

| 展示项 | 数据来源 | 说明 |
|--------|----------|------|
| 卡券种类列表 | `card_type` 表中 `status='ACTIVE'` 的记录 | 与卖家模式共享同一卡券类型定义 |
| 每种卡券的可售数量 | `order` 表中 `status='SUCCESS'` 且 `is_sold=0` 的记录按 `card_type_id` 分组统计 | 仅展示已完成回收且未售出的卡券数量 |
| 卡券图标/名称 | `card_type` 表的 `icon`、`name` 字段 | 与卖家模式一致 |

#### 3.2.2 可售数量为0的卡券种类

- **仍然展示**该卡券种类入口（保持分类完整性）
- 在卡券图标下方显示 **"暂无库存"** 标识，替代数量显示
- 用户点击进入后，在卡券列表页展示空状态提示

#### 3.2.3 热门购买推荐

| 规则 | 说明 |
|------|------|
| 推荐条件 | 可售数量 > 0 的卡券类型中，按可售数量降序取前3个 |
| 推荐信息 | 卡券类型名称 + 最低可售面值 + 对应折扣率 |
| 无可售卡券时 | 不展示"热门购买"区域，展示空状态提示文案 |

### 3.3 首页交互逻辑

| 交互动作 | 系统响应 |
|----------|----------|
| 点击卡券种类图标 | 跳转到该种类的可售卡券列表页 |
| 搜索卡券 | 按卡券名称模糊搜索，仅展示有可售库存的种类 |
| 下拉刷新 | 重新请求可售卡券数据，更新库存数量 |

## 七、数据库设计

### 7.1 新增数据表

#### 7.1.1 购买订单表 (t_buy_order)

```sql
CREATE TABLE IF NOT EXISTS `buy_order` (
  `id` INT NOT NULL AUTO_INCREMENT COMMENT '购买订单ID',
  `order_no` VARCHAR(20) NOT NULL COMMENT '购买订单号(前缀BUY)',
  `buyer_id` INT NOT NULL COMMENT '买家用户ID',
  `recycle_order_id` INT NOT NULL COMMENT '关联的回收订单ID',
  `card_type_id` INT NOT NULL COMMENT '卡券类型ID',
  `card_type_name` VARCHAR(50) NOT NULL COMMENT '卡券类型名称',
  `face_value` DECIMAL(10,2) NOT NULL COMMENT '面值金额',
  `discount_rate` DECIMAL(3,2) NOT NULL COMMENT '购买折扣率',
  `buy_price` DECIMAL(10,2) NOT NULL COMMENT '购买价格(面值×折扣率)',
  `card_no` VARCHAR(255) DEFAULT NULL COMMENT '卡号(加密,支付成功后解密展示)',
  `card_pwd` VARCHAR(255) DEFAULT NULL COMMENT '卡密(加密,支付成功后解密展示)',
  `status` VARCHAR(20) NOT NULL DEFAULT 'PENDING' COMMENT '状态: PENDING-待支付, PAID-已支付, CANCELLED-已取消',
  `pay_time` DATETIME DEFAULT NULL COMMENT '支付时间',
  `cancel_time` DATETIME DEFAULT NULL COMMENT '取消时间',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  UNIQUE KEY `uk_recycle_order_id` (`recycle_order_id`) COMMENT '一个回收订单只能被购买一次',
  KEY `idx_buyer_id` (`buyer_id`),
  KEY `idx_status` (`status`),
  KEY `idx_card_type_id` (`card_type_id`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='购买订单表';
```

**关键设计说明**：
- `recycle_order_id` 设置 **唯一索引**（`uk_recycle_order_id`），确保一个回收订单只能被购买一次，从数据库层面防止超卖
- `card_no` 和 `card_pwd` 从关联的回收订单中复制过来（加密存储），支付成功后对买家解密展示
- 购买订单号前缀为 `BUY`，与回收订单号区分

### 7.4 数据一致性保障

#### 7.4.1 防超卖机制

| 层级 | 机制 | 说明 |
|------|------|------|
| 数据库层 | `uk_recycle_order_id` 唯一索引 | 同一回收订单只能创建一个购买订单，数据库层面保证唯一性 |
| 应用层 | 乐观锁 / 分布式锁 | 创建购买订单时，先检查 `is_sold=0`，再使用事务更新 `is_sold=1`，防止并发超卖 |
| 事务保证 | 数据库事务 | 购买操作在事务中完成：创建 `buy_order` + 更新 `order.is_sold=1`，两步原子操作 |

#### 7.4.2 购买事务流程

```sql
-- 购买事务（伪代码）
BEGIN TRANSACTION;

-- 1. 检查回收订单是否可售
SELECT id, is_sold, status FROM order WHERE id = #{recycleOrderId} AND status = 'SUCCESS' AND is_sold = 0;
-- 若查询结果为空，则该卡券不可售，回滚事务

-- 2. 创建购买订单
INSERT INTO buy_order (order_no, buyer_id, recycle_order_id, card_type_id, card_type_name, face_value, discount_rate, buy_price, card_no, card_pwd, status)
VALUES (#{buyOrderNo}, #{buyerId}, #{recycleOrderId}, #{cardTypeId}, #{cardTypeName}, #{faceValue}, #{discountRate}, #{buyPrice}, #{cardNo}, #{cardPwd}, 'PENDING');

-- 3. 标记回收订单为已售出
UPDATE order SET is_sold = 1, buy_order_id = #{buyOrderId} WHERE id = #{recycleOrderId} AND is_sold = 0;
-- 若 UPDATE 影响行数为0，说明已被其他用户购买，回滚事务

COMMIT;
-- 若任何步骤失败，ROLLBACK
```

#### 7.4.3 取消购买的数据回滚

当买家取消购买订单时，需要将回收订单的 `is_sold` 状态回滚为可售：

```sql
BEGIN TRANSACTION;

-- 1. 更新购买订单状态为 CANCELLED
UPDATE buy_order SET status = 'CANCELLED', cancel_time = NOW() WHERE id = #{buyOrderId} AND status = 'PENDING';

-- 2. 回滚回收订单的售出状态
UPDATE order SET is_sold = 0, buy_order_id = NULL WHERE id = #{recycleOrderId} AND buy_order_id = #{buyOrderId};

COMMIT;
```

#### 7.4.4 支付超时自动取消

| 规则 | 说明 |
|------|------|
| 超时时间 | 购买订单创建后 30 分钟未支付则自动取消 |
| 实现方式 | 后端定时任务（每5分钟扫描一次）检查超时订单 |
| 取消后 | 购买订单状态更新为 `CANCELLED`，回收订单 `is_sold` 回滚为 `0`，卡券重新可售 |

## 十三、异常场景处理

| 异常场景 | 处理方式 |
|----------|----------|
| 卡券已被其他用户购买 | 提示"该卡券已被购买，请选择其他卡券"，刷新列表移除该卡券 |
| 并发购买同一卡券 | 后端事务保证唯一性，第二个购买请求返回错误码 `40001` |
| 支付超时 | 30分钟未支付自动取消，回收订单 `is_sold` 回滚为 `0` |
| 网络异常 | 提示"网络异常，请检查网络后重试" |
| 登录过期 | 自动跳转登录页，提示"登录已过期" |
| 未实名认证 | 弹出认证引导，认证后继续购买流程 |
| 回收订单状态变更 | 若回收订单在购买过程中被管理员标记为异常，购买订单自动取消并退款 |
| 买家取消购买 | 回收订单 `is_sold` 回滚，卡券重新上架可售 |

## 十四、非功能性要求

### 14.1 性能要求

| 指标 | 要求 |
|------|------|
| 可售卡券列表加载时间 | ≤ 2秒 |
| 购买下单响应时间 | ≤ 1秒 |
| 并发购买处理 | 支持同一卡券多人同时点击购买，仅一人成功，其余返回已被购买提示 |

### 14.2 安全要求

| 要求 | 说明 |
|------|------|
| 卡号卡密加密存储 | `buy_order` 表中 `card_no`、`card_pwd` 使用 AES 加密存储 |
| 卡号卡密按状态展示 | 仅 `PAID` 状态的订单对买家解密展示卡号卡密 |
| 防超卖 | 数据库唯一索引 + 应用层事务双重保障 |
| 防刷单 | 同一用户对同一卡券不可重复购买；限流中间件控制购买接口调用频率 |

### 14.3 数据一致性要求

| 要求 | 说明 |
|------|------|
| 购买与售出标记原子性 | 创建购买订单和标记回收订单已售出必须在同一事务中完成 |
| 取消购买与回滚原子性 | 取消购买订单和回滚回收订单售出状态必须在同一事务中完成 |
| 支付超时取消与回滚 | 定时任务取消超时订单时，同步回滚回收订单售出状态 |
